# Технический аудит проекта

## P0
### АУД-01
- **Описание:** Валидация Telegram initData не реализована на уровне зависимости `get_current_user`. В текущей имплементации авторизация строится только на проверке bearer-токена, без привязки к Telegram-сессии, что нарушает требование Backend must verify Telegram initData.
- **Файл:** `apps/backend/app/api/dependencies.py`
- **Причина:** Токен выдается без верификации initData, возможна подделка identity вне Telegram.
- **Рекомендуемое решение:** Добавить верификацию Telegram initData в `app/core/telegram.py` и привязать выдачу токена к валидному initData; добавить проверку при авторизации.

### АУД-02
- **Описание:** Лимит 3 preview применяется и к залогиненным студентам, хотя по правилам гостевой лимит должен действовать только для гостей.
- **Файл:** `apps/backend/app/services/vacancies.py:125-171`
- **Причина:** Используется единая константа `GUEST_VIEW_LIMIT` и единая логика для user и guest.
- **Рекомендуемое решение:** В блоке `user is not None` проверять роль: если student — пропускать лимит, иначе применять лимит; для guest оставить текущую логику.

### АУД-03
- **Описание:** Вакансия попадает в фид по `status == active`, без проверки `moderation_status`, что позволяет увидеть непроверенную вакансию.
- **Файл:** `apps/backend/app/services/vacancies.py:47-54`, `apps/backend/app/models/organization.py:72-77`
- **Причина:** Отсутствует 필трация по approved/moderation статусу.
- **Рекомендуемое решение:** Добавить в базовый запрос `Vacancy.moderation_status == "approved"` и статус published.

### АУД-04
- **Описание:** При создании отклика не проверяется, что вакансияプロperly модерациирована и активна.
- **Файл:** `apps/backend/app/services/applications.py:17-21`
- **Причина:** Используется только статус active, без проверки moderation.
- **Рекомендуемое решение:** Переиспользовать проверку из vacancies или добавить явную валидацию moderation_status.

### АУД-05
- **Описание:** Webhook событие дублируется при повторных вызовах confirm, нарушение идемпотентности event logging.
- **Файл:** `apps/backend/app/services/student_finance.py:196-213`
- **Причина:** При повторном вызове для уже SUCCEEDED платежа логируется `webhook_received` ещё раз.
- **Рекомендуемое решение:** Логировать `webhook_received` только один раз или хранить флаг processed_at.

### АУД-06
- **Описание:** Фронтенд определяет доступ по просмотру через localStorage, что небезопасно: клиент может сбросить данные и обойти лимит.
- **Файл:** `apps/web/src/lib/guest.ts`, `apps/backend/app/services/vacancies.py`
- **Причина:** Frontend guest counter не trusted, backend лимит уже есть, но фронт дублирует логику неправильно.
- **Рекомендуемое решение:** Убрать клиентский лимит и доверять только бэкенду; использовать фронтенд только для UX.

### АУД-07
- **Описание:** Нет проверки, что вакансия принадлежит компании HR при доступе к заявкам и деталям.
- **Файл:** `apps/backend/app/services/applications.py:112-143`
- **Причина:** HR может видеть заявки по любым вакансиям, где он hr_user_id, но без привязки к компании.
- **Рекомендуемое решение:** Добавить проверку, что vacancy.company_id == hr_user.hr_profile.company_id.

## P1
### АУД-08
- **Описание:** Отсутствует пагинация в листинге вакансий и заявок, риск падения производительности при росте данных.
- **Файл:** `apps/backend/app/api/routes/vacancies.py:30-53`, `apps/backend/app/services/applications.py:94-124`
- **Причина:** Используется `list(...).all()` без cursor/limit.
- **Рекомендуемое решение:** Добавить пагинацию через `limit/offset` или cursor-based.

### АУД-09
- **Описание:** Баланс вычисляется каждый раз как сумма транзакций, что медленно при большом объеме.
- **Файл:** `apps/backend/app/services/student_finance.py:59-61`
- **Причина:** Отсутствует denormalized cached_balance на User.
- **Рекомендуемое решение:** Ввести поле `cached_balance` на User с обновлением в транзакции.

### АУД-10
- **Описание:** Нет индексов на часто используемые фильтры, что приведет к медленным запросам.
- **Файл:** `apps/backend/app/models/organization.py`, `apps/backend/app/models/user.py`, `apps/backend/app/models/finance.py`
- **Причина:** Отсутствуют индексы на `Vacancy.status`, `Vacancy.expires_at`, `Vacancy.moderation_status`, `Payment.user_id`, `BalanceTransaction.user_id`.
- **Рекомендуемое решение:** Добавить индексы в Alembic миграции.

### АУД-11
- **Описание:** Возможна повторная отправка отклика в случае гонки, идемпотентность не гарантирована на уровне БД.
- **Файл:** `apps/backend/app/services/applications.py:38-46`
- **Причина:** Используется SELECT + INSERT, без unique constraint.
- **Рекомендуемое решение:** Добавить UniqueConstraint на `(vacancy_id, student_user_id)`.

### АУД-12
- **Описание:** HR может создать вакансию с невалидными/несуществующими данными компании.
- **Файл:** `apps/backend/app/services/hr_vacancies.py` (предположительно)
- **Причина:** Нет явной проверки что HR принадлежит компании.
- **Рекомендуемое решение:** Добавить проверку ownership компании при создании вакансии.

## P2
### АУД-13
- **Описание:** Дублирование логики контактов в serialize_application_for_hr.
- **Файл:** `apps/backend/app/services/visibility.py:54-81`
- **Причина:** Повторяются проверки can_hr_view_student_contacts.
- **Рекомендуемое решение:** Вынести общую валидацию в отдельную функцию.

### АУД-14
- **Описание:** Отсутствует ограничение количества одновременно активных подписок у одного студента.
- **Файл:** `apps/backend/app/services/student_finance.py:74-88`
- **Причина:** get_latest_subscription возвращает последнюю, но не проверена уникальность активной.
- **Рекомендуемое решение:** Добавить UniqueConstraint или проверку при создании.

### АУД-15
- **Описание:** Нет проверки формата телефона/email на бэкенде.
- **Файл:** `apps/backend/app/models/user.py`
- **Причина:** Поля phone/email без regex ограничений.
- **Рекомендуемое решение:** Добавить валидаторы или Pydantic схемы.

## P3
### АУД-16
- **Описание:** Нет аудита изменений ролей пользователей.
- **Файл:** `apps/backend/app/api/routes/admin.py`
- **Причина:** Изменение роли не логируется событием.
- **Рекомендуемое решение:** Добавить event logging при PATCH /api/admin/users/{id}/role.

### АУД-17
- **Описание:** Отсутствует возможность удаления жалоб, только resolve.
- **Файл:** `apps/backend/app/models/misc.py`, `apps/backend/app/api/routes/admin.py`
- **Причина:** Нет эндпоинта DELETE complaints.
- **Рекомендуемое решение:** Добавить админский эндпоинт удаления complaint.

### АУД-18
- **Описание:** Нет проверки длины текстовых полей на бэкенде.
- **Файл:** `apps/backend/app/schemas/*`
- **Причина:** В схемах нет ограничений max_length.
- **Рекомендуемое решение:** Добавить Field(max_length=...) в Pydantic схемах.
