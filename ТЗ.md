# ТЗ для Codex: Telegram-бот + Telegram Mini App для сервиса студенческих вакансий

## 1. Контекст проекта

Нужно разработать MVP сервиса студенческих вакансий.

Проект состоит из:

1. Telegram-бота.
2. Telegram Mini App.
3. Backend API.
4. Базы данных.
5. Админской логики.
6. Интеграции оплаты через ЮKassa.
7. Базовой AI-модерации вакансий.

Сервис работает как закрытая платформа вакансий для студентов и HR.

Студент может бесплатно посмотреть только 3 вакансии. После этого для продолжения просмотра и откликов он должен зарегистрироваться и активировать доступ на месяц через первичное пополнение баланса.

HR может зарегистрироваться только через админа или HR-реферальный код. HR создаёт вакансии, оплачивает размещение, получает отклики студентов и видит контакты студента только после принятия отклика.

Студент никогда не видит контакты HR, даже после отклика.

## 3. Главные роли

В системе должны быть роли:

1. `student` — студент.
2. `hr` — работодатель/HR.
3. `admin` — администратор.
4. `moderator` — модератор, можно заложить, но на MVP можно не использовать отдельно.
5. `support` — поддержка, можно заложить, но на MVP можно не использовать отдельно.

По умолчанию новый пользователь получает роль `student`.

HR-доступ нельзя получить самостоятельно. HR создаётся:

* админом;
* через валидный HR referral code.

## 4. Основная логика студента

### 4.1 Первый вход

Студент открывает Telegram-бота.

Бот показывает:

* приветствие;
* кнопку “Открыть вакансии”;
* кнопку “Как это работает”;
* кнопку “Зарегистрироваться”.

При открытии Mini App студент попадает в ленту вакансий.

Если пользователь ещё не зарегистрирован и не оплатил доступ, он может посмотреть только 3 вакансии.

Нужно считать просмотры вакансий гостем.

### 4.2 Ограничение 3 вакансии

Логика:

* незарегистрированный пользователь может открыть максимум 3 карточки вакансий;
* после 3 просмотров лента блокируется paywall-экраном;
* откликнуться без регистрации нельзя;
* если пользователь нажимает “Откликнуться” без регистрации, показать экран регистрации и оплаты.

Текст paywall:

“Ты посмотрел 3 бесплатные вакансии. Чтобы смотреть больше и откликаться, зарегистрируйся и активируй доступ на месяц.”

Кнопки:

* “Зарегистрироваться”
* “Активировать доступ”
* “Как это работает”

### 4.3 Регистрация студента

Поля профиля студента:

* имя;
* фамилия;
* телефон;
* Telegram username;
* email;
* учебное заведение;
* курс;
* специальность / направление;
* интересующий тип работы;
* желаемый график;
* район / формат работы;
* опыт, кратко;
* согласие с пользовательским соглашением;
* согласие на обработку персональных данных.

На MVP не собирать банковские данные студента.

### 4.4 Первичное пополнение баланса

Регистрация считается завершённой только после первичного пополнения баланса минимум на стоимость месячного тарифа.

Логика:

* тариф на месяц, например `350 RUB`;
* пользователь может пополнить баланс на 350, 700, 1050 или свою сумму;
* после успешной оплаты списать стоимость тарифа;
* активировать доступ на 30 дней;
* остаток оставить на внутреннем балансе.

Пример:

Студент пополнил 700 ₽.
Списываем 350 ₽ за первый месяц.
Остаток 350 ₽ остаётся на балансе.

### 4.5 Возврат денег

В интерфейсе нужно показать текст:

“Возврат остатка баланса возможен через заявку в поддержку. Комиссия платёжной системы и расходы на обработку платежа могут быть удержаны.”

Сделать кнопку:

* “Запросить возврат”

На MVP заявка на возврат создаёт support ticket, автоматический возврат делать не нужно.

### 4.6 Доступ после оплаты

Если у студента активен тариф:

* он может смотреть всю ленту вакансий;
* открывать карточки;
* пользоваться фильтрами;
* откликаться;
* видеть историю откликов;
* получать уведомления.

Если тариф истёк:

* показывать paywall;
* отклики запрещены;
* можно дать снова только 3 preview-просмотра или полностью закрыть доступ. Для MVP использовать вариант: после истечения тарифа доступ к откликам закрыт, лента снова ограничена 3 вакансиями.

## 5. Логика откликов

### 5.1 Как студент откликается

На карточке вакансии есть кнопка “Откликнуться”.

При нажатии:

1. Проверить активен ли доступ.
2. Проверить заполнен ли профиль.
3. Создать отклик.
4. Показать сообщение “Отклик отправлен”.
5. Вернуть студента на главную страницу ленты.

Студент не видит контакты HR.

### 5.2 Что видит студент после отклика

В истории откликов студент видит:

* вакансию;
* компанию;
* дату отклика;
* статус.

Статусы отклика:

* `sent` — отправлен;
* `viewed` — просмотрен HR;
* `accepted` — HR принял отклик;
* `rejected` — HR отклонил отклик;
* `complaint` — по отклику есть жалоба;
* `closed` — вакансия закрыта.

Даже если статус `accepted`, студент не видит контакты HR.

Можно отправить студенту уведомление:

“HR заинтересовался твоим откликом. Если ты указал актуальные контакты, с тобой свяжутся.”

### 5.3 Что видит HR до принятия отклика

HR видит карточку отклика без контактных данных.

Показать:

* имя студента;
* учебное заведение;
* курс;
* специальность;
* желаемый график;
* опыт;
* комментарий к отклику, если есть;
* дата отклика.

Не показывать:

* телефон;
* email;
* Telegram username.

Кнопки:

* “Принять”
* “Отклонить”
* “Пожаловаться”

### 5.4 Что видит HR после принятия отклика

После нажатия “Принять”:

* статус отклика меняется на `accepted`;
* HR получает контакты студента:

  * телефон;
  * Telegram username;
  * email.

Студент получает уведомление, что HR заинтересовался.

## 6. Логика HR

### 6.1 Регистрация HR

HR не может просто сам зарегистрироваться как HR.

Варианты:

1. Админ вручную выдаёт пользователю роль `hr`.
2. Пользователь вводит валидный HR referral code.
3. После проверки кода пользователь становится HR.

HR-профиль:

* название компании;
* ИНН, если есть;
* имя контактного лица;
* должность;
* телефон;
* email;
* Telegram username;
* описание компании;
* логотип, если есть;
* статус верификации.

Статусы HR:

* `pending`;
* `active`;
* `blocked`.

### 6.2 Создание вакансии

HR создаёт вакансию через Mini App или форму, открываемую из бота.

Поля вакансии:

* название;
* компания;
* категория;
* тип занятости:

  * подработка;
  * смена;
  * стажировка;
  * постоянная работа;
  * проектная работа;
* график;
* оплата;
* адрес / район;
* формат:

  * офлайн;
  * онлайн;
  * гибрид;
* описание;
* обязанности;
* требования;
* условия;
* нужен ли опыт;
* фото, опционально;
* срок публикации;
* скрытые контакты HR;
* дата создания.

После заполнения HR должен оплатить размещение.

### 6.3 Оплата размещения

Для MVP заложить тарифы как конфиг, чтобы их можно было менять без переписывания логики.

Минимальные тарифы:

* базовое размещение на 30 дней;
* продвижение на 7 дней.

Тарифы хранить в таблице `tariffs`.

Пример:

* vacancy_basic_30d = 149 RUB;
* promotion_color_7d = 199 RUB;
* promotion_xl_7d = 299 RUB;
* promotion_top_7d = 399 RUB.

После оплаты:

* вакансия получает статус `moderation`;
* запускается AI-модерация;
* если модерация успешна — статус `active`;
* если сомнительно — статус `manual_review`;
* если запрещено — статус `rejected`.

### 6.4 Продвижение вакансии

Виды продвижения:

1. Выделение цветом.
2. XL-карточка.
3. Поднятие в топ.

Для MVP достаточно реализовать:

* `is_promoted`;
* `promotion_type`;
* `promotion_started_at`;
* `promotion_expires_at`.

В ленте продвинутые вакансии должны визуально отличаться и ранжироваться выше обычных.

## 11. AI-модерация вакансий

После создания и оплаты вакансии запустить AI-модерацию.

Модерация должна проверять:

* скам;
* казино/ставки;
* интим/18+;
* финансовые пирамиды;
* сетевой маркетинг;
* подозрительно высокий доход без условий;
* вакансии с предоплатой от студента;
* вакансии без понятного работодателя;
* опасные или незаконные условия;
* дискриминационные формулировки;
* просьбы передать документы неизвестным лицам.

Результаты:

* `approved`;
* `manual_review`;
* `rejected`.

На MVP можно сделать простую функцию-заглушку с правилами, а затем заменить на реальный AI-вызов.

## 14. Требования к безопасности

1. Проверять Telegram initData на backend.
2. Не доверять данным с frontend.
3. Все действия HR проверять по роли.
4. Все действия admin проверять по роли.
5. Контакты студента отдавать HR только если отклик принят.
6. Контакты HR не отдавать студенту никогда.
7. Webhook ЮKassa проверять.
8. Все платежи должны быть идемпотентны.
9. Не активировать подписку без подтверждённой оплаты.
10. Логировать критические действия.

## 16. Тестовые сценарии

Проверить:

1. Гость видит 3 вакансии.
2. После 3 вакансий появляется paywall.
3. Гость не может откликнуться.
4. Студент регистрируется.
5. Студент пополняет баланс.
6. После оплаты активируется доступ.
7. Активный студент может откликнуться.
8. После отклика студент возвращается на главную.
9. Студент не видит контакты HR.
10. HR видит отклик без контактов.
11. HR принимает отклик.
12. HR видит контакты студента.
13. Студент получает уведомление, что HR заинтересовался.
14. HR создаёт вакансию.
15. HR оплачивает размещение.
16. Вакансия проходит модерацию.
17. Вакансия появляется в ленте.
18. Админ может выдать HR-доступ.
19. Админ может заблокировать пользователя.
20. Webhook оплаты не создаёт двойное начисление при повторном вызове.

## 17. Важное ограничение

Не делать лишнего сверх MVP.

Не реализовывать на первом этапе:

* сложную карту;
* сложный рейтинг;
* полноценную CRM уровня amoCRM;
* автоматические возвраты;
* сложные HR-тарифы;
* мобильное приложение вне Telegram;
* полноценную рекомендательную систему.

Сначала нужен рабочий end-to-end flow:

HR создал и оплатил вакансию → студент увидел → студент оплатил доступ → студент откликнулся → HR принял → HR получил контакты.
